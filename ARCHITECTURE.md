# 报文发送系统架构设计

## 整体架构

```
┌─────────────────┐
│   Web浏览器      │
│  (Django前端)    │
└────────┬────────┘
         │ HTTP/HTTPS
         │
┌────────▼─────────────────────────┐
│      Django后端服务器            │
│  - 接收前端请求                  │
│  - 与代理程序通信                │
│  - 数据转换和验证                │
└────────┬─────────────────────────┘
         │ HTTP API
         │
┌────────▼─────────────────────────┐
│   报文发送代理程序                │
│  (运行在远程主机上)               │
│  - Flask HTTP API服务器          │
│  - Scapy报文构造和发送           │
│  - 实时统计                      │
└────────┬─────────────────────────┘
         │ 原始报文
         │
┌────────▼─────────────────────────┐
│      网络接口                     │
│  (网卡)                          │
└──────────────────────────────────┘
```

## 方案对比

### 方案1: SSH方式（当前实现）
**优点：**
- 无需额外安装代理程序
- 使用系统自带工具

**缺点：**
- 报文构造能力有限
- 无法精确控制所有字段
- 性能较差（命令执行开销）
- 不支持实时统计

### 方案2: 代理程序方式（推荐）⭐
**优点：**
- ✅ 精确控制报文的每个字节
- ✅ 支持MAC/IP/TCP/UDP所有层参数
- ✅ 高性能（本地发送）
- ✅ 实时统计（发送速率、带宽）
- ✅ 支持报文变化功能
- ✅ 支持连续发送和批量发送

**缺点：**
- 需要在远程主机安装代理程序
- 需要管理员权限（发送原始报文）

## 推荐方案：代理程序

### 技术栈

**代理程序：**
- Python 3.7+
- Flask (HTTP API服务器)
- Scapy (报文构造和发送)
- Npcap (Windows) / libpcap (Linux)

**Django后端：**
- requests (与代理程序通信)
- 数据验证和转换

**前端：**
- Bootstrap 5
- JavaScript (实时通信)

### 部署架构

```
远程主机1 (Windows 7)
├── 代理程序 (packet_agent.py)
│   └── 监听: http://192.168.1.100:8888
│
远程主机2 (Linux)
├── 代理程序 (packet_agent.py)
│   └── 监听: http://192.168.1.101:8888
│
Django服务器
├── 接收前端请求
├── 转发到对应代理程序
└── 返回结果给前端
```

### 工作流程

1. **前端** → 用户填写代理程序URL和报文配置
2. **Django后端** → 验证配置，转发到代理程序
3. **代理程序** → 构造报文，发送到网络
4. **代理程序** → 返回统计信息
5. **Django后端** → 转发统计信息给前端
6. **前端** → 实时显示发送状态和统计

### 安全考虑

1. **网络隔离**：代理程序只监听内网IP
2. **API认证**：可选的API密钥认证
3. **HTTPS**：生产环境使用HTTPS
4. **防火墙**：限制代理程序端口访问

### 安装步骤

#### Windows 7
1. 安装 Npcap: https://nmap.org/npcap/
2. 安装 Python 3.7+
3. 安装依赖: `pip install flask flask-cors scapy`
4. 运行: `python packet_agent.py --host 0.0.0.0 --port 8888`

#### Linux
1. 安装依赖: `sudo apt-get install python3-pip libpcap-dev`
2. 安装Python包: `pip3 install flask flask-cors scapy`
3. 运行（需要root）: `sudo python3 packet_agent.py --host 0.0.0.0 --port 8888`

## 报文配置参数

### MAC层
- 源MAC地址
- 目的MAC地址

### IP层
- 源IP地址
- 目的IP地址
- TTL
- TOS

### TCP层
- 源端口
- 目的端口
- 序列号
- ACK序号
- 标志位 (SYN/ACK/FIN/RST/PSH/URG)
- 窗口大小
- 紧急指针

### 数据层
- 载荷内容（十六进制或ASCII）

### 报文变化
- 支持字段递增、固定、随机等变化模式
- 可配置变化次数

### 发送控制
- 发送次数（固定次数或连续发送）
- 发送间隔（延迟或无延迟）

## API接口设计

### 代理程序API

1. `GET /api/interfaces` - 获取网卡列表
2. `POST /api/send_packet` - 发送报文
3. `GET /api/statistics` - 获取统计
4. `POST /api/stop` - 停止发送
5. `GET /api/health` - 健康检查

### Django API

1. `POST /api/agent/interfaces/` - 获取代理程序网卡列表
2. `POST /api/agent/send_packet/` - 通过代理程序发送报文
3. `POST /api/agent/statistics/` - 获取代理程序统计
4. `POST /api/agent/stop/` - 停止代理程序发送

## 总结

**推荐使用代理程序方案**，因为：
1. 可以满足精确的报文构造需求
2. 性能更好
3. 功能更完整
4. 易于扩展和维护

SSH方式可以作为备用方案，用于简单的ping测试等场景。
